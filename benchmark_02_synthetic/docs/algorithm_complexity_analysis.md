# Algorithm Time And Space Complexity

This note analyzes the four search algorithms used in the V2 evaluation:

1. Algorithm 1: the current app search evaluator;
2. Algorithm 2: the external English fast search algorithm;
3. Algorithm 3: the V2 master rank-fusion algorithm;
4. Algorithm 4: Algorithm 2 plus a lightweight family-level rescue and safety
   layer.

The goal is not to pretend the algorithms are one-line textbook searches. Each
algorithm has two phases:

- a preparation phase that builds normalized records and indexes;
- a per-query phase that retrieves candidates, scores them, sorts them, and
  returns ranked results.

## Symbols

| symbol | meaning |
| --- | --- |
| `N` | number of catalog product records. In the current catalog this is about 25k records. |
| `F` | number of unique commercial families after grouping. `F <= N`. |
| `L` | average compact commercial-name length. |
| `Q` | compact query length. |
| `T` | number of query tokens. |
| `C_current` | number of candidate records generated by the current app indexes for one query. |
| `C_external` | number of candidate records generated by the external algorithm indexes for one query. |
| `K_child` | number of child results requested by the master from each child algorithm. In V2 this is `40`. |
| `M` | number of merged master candidates. `M <= 2 * K_child`, before duplicate family merging. |
| `C_rescue` | number of family candidates generated by Algorithm 4's rescue buckets for one query. |
| `R` | number of merged Algorithm 4 candidates. `R <= C_external + C_rescue`, then capped for output. |

The important practical point is that `C_current` and `C_external` are normally
much smaller than `N`. Worst-case buckets can still be large, but the indexes
are designed to avoid scoring every catalog row for every query.

## Summary Table

| algorithm | preprocessing time | preprocessing space | per-query time | per-query extra space |
| --- | ---: | ---: | ---: | ---: |
| Algorithm 1 | `O(N * L^2)` worst case | `O(N * L^2)` worst case | `O(C_current * Q * L + C_current log C_current)` | `O(C_current)` |
| Algorithm 2 | `O(N * L^2)` worst case | `O(N * L^2)` worst case | `O(Q^2 + C_external * Q * L + C_external log C_external)` | `O(C_external)` |
| Algorithm 3 | Algorithm 1 + Algorithm 2 preprocessing | Algorithm 1 + Algorithm 2 indexes | Algorithm 1 query + Algorithm 2 query + `O(M log M)` | Algorithm 1 + Algorithm 2 query memory + `O(M)` |
| Algorithm 4 | Algorithm 2 preprocessing + `O(F * L^2)` family rescue index | Algorithm 2 index + `O(F * L^2)` rescue index | Algorithm 2 query + optional context-clean Algorithm 2 query + `O(C_rescue * Q * L + R log R)` | `O(C_external + C_rescue)` |

The `L^2` term mainly comes from delete-key indexes. The algorithms generate
one-delete or two-delete variants to recover missing-letter queries. Because
medicine names are short in practice and the delete depth is capped, this is
manageable.

## 1. Current App Search Evaluator

### What It Does

The current app evaluator is a Python port of the browser scoring rules. It
normalizes catalog records, builds indexes, generates a candidate set for each
query, scores only those candidates, then sorts the scored candidates.

It builds indexes for:

- exact normalized names;
- compact names;
- prefixes;
- trigrams;
- tokens;
- consonant skeletons;
- phonetic keys;
- delete keys;
- numbers and route hints;
- short-prefix danger analysis.

### Preprocessing Complexity

For each catalog record, the algorithm creates normalized fields, compact
fields, skeletons, phonetic keys, grams, prefixes, and delete variants.

Most fields are linear in the record length, so they are `O(L)`.

Delete variants are heavier. For records with length up to 7, it creates
one-delete variants. For longer records, it creates up to two-delete variants.
That gives worst-case `O(L^2)` variants per record.

So:

```text
preprocessing time  = O(N * L^2)
preprocessing space = O(N * L^2)
```

In practice this is closer to linear because `L` is small and delete depth is
capped.

### Per-Query Complexity

Per query, the algorithm:

1. normalizes the query: `O(Q)`;
2. looks up exact, prefix, gram, token, skeleton, phonetic, and delete buckets;
3. builds a candidate set of size `C_current`;
4. scores each candidate;
5. sorts scored candidates.

The expensive scoring operation is bounded edit distance against the candidate
base group. That is approximately `O(Q * L)` per candidate in the worst case,
although the implementation exits early when the edit distance exceeds the
allowed threshold.

So:

```text
per-query time  = O(C_current * Q * L + C_current log C_current)
per-query space = O(C_current)
```

If a query lands in a very broad bucket, `C_current` can approach `N`, making
the worst case:

```text
worst-case per-query time = O(N * Q * L + N log N)
```

But that is not the normal case. The evaluator uses rare grams, capped buckets,
and route/number restrictions to keep the candidate set smaller.

## 2. External English Fast Algorithm

### What It Does

The external English fast algorithm is more typo-focused. It builds many indexes
and then scores candidates using several similarity functions:

- exact and compact match;
- alias match;
- prefix and suffix similarity;
- contains similarity;
- subsequence score;
- consonant skeleton similarity;
- phonetic similarity;
- weighted n-gram Jaccard;
- token score;
- normal Damerau-Levenshtein edit similarity;
- weighted Damerau-Levenshtein edit similarity.

This makes it stronger on noisy commercial-name typos, but each candidate score
is heavier than the current app scorer.

### Preprocessing Complexity

Like the current evaluator, it creates normalized records, grams, prefixes,
suffixes, skeletons, phonetic keys, token statistics, IDF values, and delete
keys.

The delete-key generation again dominates worst-case preprocessing:

```text
preprocessing time  = O(N * L^2)
preprocessing space = O(N * L^2)
```

It also stores token and gram document-frequency maps:

```text
additional space = O(number of unique tokens + number of unique grams)
```

That is still bounded by the total catalog text size.

### Per-Query Complexity

Per query, it:

1. normalizes and classifies the query: `O(Q)`;
2. builds query grams and delete keys: up to `O(Q^2)`;
3. collects candidates from exact, prefix, suffix, token, n-gram, skeleton,
   phonetic, and delete-key indexes;
4. computes a full feature vector for each candidate;
5. groups duplicate product rows by family;
6. sorts grouped results.

The expensive part is edit similarity. It computes both normal and weighted
Damerau-Levenshtein similarity, each `O(Q * L)` per candidate.

So:

```text
per-query time = O(Q^2 + C_external * Q * L + C_external log C_external)
```

The `C_external log C_external` term comes from grouping and sorting candidate
families after scoring.

Per-query extra memory:

```text
per-query space = O(C_external)
```

The edit-similarity cache is also proportional to the number of candidate pairs
scored for that query, so it is included in `O(C_external)`.

Worst case, if candidate generation becomes too broad:

```text
worst-case per-query time = O(Q^2 + N * Q * L + N log N)
```

Again, normal runtime is lower because the external algorithm uses rare n-grams
and capped buckets.

## 3. Master V2 Rank-Fusion Algorithm

### What It Does

The master algorithm does not search the raw catalog directly. It runs both
child algorithms, asks each for up to `40` results, merges those results by
commercial family, calculates a fusion score, applies safety gates, and returns
the final ranking.

The master is therefore a fusion layer:

```text
current app results + external fast results -> merged family candidates -> final ranking
```

### Preprocessing Complexity

The master prepares both child algorithms:

```text
master preprocessing time  = current preprocessing + external preprocessing
master preprocessing space = current index space + external index space
```

Using the simplified worst-case terms:

```text
preprocessing time  = O(N * L^2)
preprocessing space = O(N * L^2)
```

The constant factor is larger than either child alone because both index sets
exist at the same time.

### Per-Query Complexity

For every query, the master:

1. runs current app search;
2. runs external fast search;
3. collects up to `K_child = 40` results from each;
4. merges duplicate families;
5. computes fusion score for each merged candidate;
6. sorts the merged candidates;
7. applies response-status and clarification logic.

The child searches dominate the runtime:

```text
master per-query time =
  current per-query time
  + external per-query time
  + O(M log M)
```

Because `M <= 80`, the master-only part is tiny:

```text
M <= 2 * K_child = 80
O(M log M) is effectively constant in this implementation
```

So the realistic statement is:

```text
master per-query time is approximately the cost of running both child searches.
```

Per-query extra memory:

```text
master per-query space =
  current query memory
  + external query memory
  + O(M)
```

Since `M` is capped, master-specific memory is small.

### Why Master Is Slower But Better

The master intentionally pays extra time because it executes both child
algorithms. The benefit is that it can combine strengths:

- current app is safer around short prefixes, route/form context, and
  clarification behavior;
- external fast is stronger on typo-heavy commercial-name recovery;
- master uses rank fusion and safety gates to improve retrieval without allowing
  unsafe confident top-1 answers.

In the V2 full run:

| evaluation | rows | runtime |
| --- | ---: | ---: |
| current + external evaluation | 115,000 | 152.44 seconds |
| master evaluation | 115,000 | 161.88 seconds |

That runtime difference is consistent with the complexity model: the master
does more work per query, but its extra fusion layer is small compared with the
two child searches.

## 4. Algorithm 4 Fast Rescue Search

### What It Does

Algorithm 4 was created because Algorithm 3's improvement was useful but its
deployment shape was heavy: every query ran both Algorithm 1 and Algorithm 2.
Algorithm 4 keeps Algorithm 2 as the only full catalog search pass, then adds a
smaller rescue layer over unique commercial families plus a targeted
context-clean pass for strength/form noise.

The rescue layer is built for the failure patterns found in the manually written
cases:

- one or two missing/inserted middle letters;
- visually or phonetically similar substitutions;
- same-prefix/same-suffix false positives;
- exact-looking fragment traps that should require clarification;
- bounded general rescue without hard-coded manual-case aliases.

### Preprocessing Complexity

Algorithm 4 prepares Algorithm 2 normally, then builds a family-level rescue
index. The rescue index is over unique commercial families, not every product
row, so `F <= N`.

The rescue index stores exact keys, prefixes, suffixes, rare character n-grams,
skeleton keys, phonetic keys, delete keys, length buckets, first-character
buckets, prefix-risk counts, and family warning metadata.

The delete-key generation is again the largest term:

```text
rescue preprocessing time  = O(F * L^2)
rescue preprocessing space = O(F * L^2)
```

So Algorithm 4 preprocessing is:

```text
Algorithm 4 preprocessing time =
  Algorithm 2 preprocessing + O(F * L^2)

Algorithm 4 preprocessing space =
  Algorithm 2 index space + O(F * L^2)
```

Because `F` is smaller than `N`, this is materially cheaper than preparing a
second full product-row search stack like Algorithm 3 does.

### Per-Query Complexity

For every query, Algorithm 4:

1. runs Algorithm 2 once;
2. optionally runs a cleaned-context Algorithm 2 query when the input contains
   strength, unit, form, route, or similar noise;
3. runs the rescue layer only when Algorithm 2 has no result or is uncertain
   enough to justify bounded family rescue;
4. gathers rescue family candidates from exact, prefix, suffix, rare n-gram,
   skeleton, phonetic, delete-key, and optional length buckets when that gate
   opens;
5. scores only those rescue families;
6. merges Algorithm 2, context-clean, and rescue candidates by compact family
   key;
7. sorts the merged list and applies conservative safety gates.

The per-query term is:

```text
Algorithm 4 per-query time =
  Algorithm 2 query time
  + optional Algorithm 2 context-clean query time
  + O(R log R)
  + optional O(C_rescue * Q * L) when rescue is gated on
```

`C_rescue` is bounded by bucket selection and partitioned prefiltering: 45 core
families plus a two-stage 45-to-15 edge-rescue shortlist. In the latest 6,000-row
sample run, Algorithm 4's average candidate pool was 23.72,
compared with Algorithm 2's 17.97, Algorithm 3's 37.88, and Algorithm 1's
186.83. In the latest full 115,000-row run, Algorithm 4's average candidate
pool was 23.47, compared with Algorithm 2's 17.86 and Algorithm 3's 37.69.

The latest full evaluator run completed 115,000 Algorithm 4 cases in 292.95
seconds with 8 workers. The standalone times are not directly comparable to
Algorithm 3's older run because host load and harness versions differ. The OCR
microbenchmark measured 15.28 ms per fair unique query, versus 13.31 ms before
the edge rescue was added.

### Why It Is Cheaper Than Algorithm 3

Algorithm 3 pays for both child query paths:

```text
Algorithm 3 query ~= Algorithm 1 query + Algorithm 2 query
```

Algorithm 4 pays for Algorithm 2 plus a gated family rescue pass:

```text
Algorithm 4 query ~= Algorithm 2 query + optional context cleanup + bounded family rescue
```

That makes Algorithm 4 closer to Algorithm 2 than to Algorithm 3 in
architecture. The current implementation improves Algorithm 2's safety and
retrieval, but after broadening the rescue pass it should be treated as an
algorithmic experiment, not as a proven lower-latency replacement for Algorithm
3.

## Practical Interpretation

For a single query, the important driver is not the full catalog size `N`.
The important driver is candidate-set size:

```text
C_current and C_external matter more than N during normal search.
```

The indexes move the system from:

```text
score every catalog row
```

to:

```text
score only rows that share useful exact, prefix, token, n-gram, skeleton,
phonetic, or delete-key evidence.
```

That is why the system can evaluate 115,000 generated test cases in minutes
instead of scanning all catalog rows with heavy edit-distance logic for every
case.

## One-Sentence Version

Algorithm 1 and Algorithm 2 both use `O(N * L^2)` preprocessing indexes to make
per-query search depend mostly on candidate-set size. Algorithm 3 improves
behavior by running both child searches. Algorithm 4 keeps only Algorithm 2's
full search path and adds a smaller `O(F * L^2)` family rescue index; after the
recommendation pass it recovers one extra manual case, remains safe, and has a
smaller candidate pool than Algorithm 3, but its current wall-clock benchmark is
slower because rescue is broader.
