# Testing Dataset V2 Data Generation And Dataset Description

## One-Sentence Problem

Generate a realistic commercial-name medical-search benchmark where every case is produced by deterministic code, every planned category is represented, most cases are hard rather than exact-match smoke tests, and every output row carries enough provenance to explain exactly why it exists.

## Why This Dataset Exists

The earlier commercial-name suite was large and useful, but the new work needed a clearer structure:

1. Every test case must be generated by a script, not manually copied into the dataset.
2. Every category must map back to a named generator function.
3. Safety cases must be first-class, not accidental side effects of fuzzy typo tests.
4. The distribution must be realistic enough that a search algorithm cannot look good by only passing exact matches.
5. The final report must allow debugging by category and by detailed error type.

This dataset is therefore a benchmark for commercial-name search behavior, not only typo correction. It tests retrieval, ambiguity handling, no-match handling, and unsafe confident top-1 behavior.

## Important Total-Count Decision

The attached plan headline said approximately 120,000 cases. The explicit category rows in that same plan sum to 115,000 cases. The generator follows the explicit per-category targets because they are auditable and because inventing an undocumented 5,000-row category would hide a design decision.

This is recorded in `data/generation_summary.json`:

| field | value |
| --- | ---: |
| `declared_plan_total_cases` | 120,000 |
| `validated_total_cases_from_category_rows` | 115,000 |
| `total_cases` | 115,000 |

## Dataset Schema

The generator writes the legacy evaluation fields first and then adds provenance fields. This makes the file compatible with older tooling while proving every row came from a specific generator.

| column | type | values | meaning |
| --- | --- | --- | --- |

| `error_type` | string | generated by each category function | Detailed mutation label, usually including operation and position. |
| `category` | string | one of 34 planned categories | The high-level test category. |
| `difficulty` | enum | `EASY`, `MEDIUM`, `HARD`, `EXTREME` | Difficulty assigned from exact per-category quotas. |
| `danger` | enum | `SAFE`, `CAUTION`, `DANGEROUS` | Medical safety risk label. |
| `collision_with` | string | empty or semicolon-separated family names | Other catalog names that collide with this query/prefix. Capped for readability. |
| `notes` | string | generated explanation | Human-readable mutation note, plan category number, plan description, source id, and generator function. |
| `scope` | enum | `inside`, `semi_outside`, `safety`, `smoke` | Evaluation scope. |
| `tier` | enum-like string | `tier_1_product`, `tier_2_safety`, `tier_3_algorithmic`, `tier_4_smoke` | Plan tier. |
| `category_number` | integer string | `1` through `34` | Plan row number. |
| `expected_behavior` | enum | `match`, `ambiguous`, `no_match` | Behavior that counts as safe/successful for the evaluator. |
| `generator_script` | string | `benchmark_02_synthetic/generate_dataset.py` | Script that generated the row. |
| `generator_function` | string | one of the 34 category generator names | Exact function responsible for the row. |
| `source_candidate_id` | string | catalog candidate id or empty | Source catalog row identifier when a source drug exists. |
| `source_base_group` | string | catalog base group or empty | Source commercial family before mutation. |

## Expected Behavior Semantics

| value | expected search behavior | metric interpretation |
| --- | --- | --- |
| `match` | The expected commercial family should appear in the top 20. A confident top-1 is acceptable only if it is correct and not unsafe. | Retrieval metrics such as Hit@1/5/10/20 are meaningful. Behavior success requires top-20 retrieval. |
| `ambiguous` | The system should show candidates but avoid a confident automatic answer. | Behavior success requires results plus non-confident/clarifying behavior. |
| `no_match` | The system should avoid a confident drug match. | Behavior success requires no results or a non-confident/clarifying response. |

## Difficulty Distribution

The plan required more than 60% hard cases. The generated dataset has 84,400 hard or extreme cases, which is 73.39%.

| difficulty | rows |
| --- | ---: |
| `EASY` | 6,900 |
| `MEDIUM` | 23,700 |
| `HARD` | 53,900 |
| `EXTREME` | 30,500 |
| `HARD + EXTREME` | 84,400 |
| `HARD + EXTREME ratio` | 73.39% |

The generator assigns difficulty after selecting category rows. This matters because deterministic sampling can change the exact set of cases; assigning the quotas after selection guarantees the final counts match the plan exactly.

## Danger Distribution

| danger | rows | intended meaning |
| --- | ---: | --- |
| `SAFE` | 80,931 | Expected typo or formatting problem where a wrong answer is less likely to be directly dangerous. |
| `CAUTION` | 11,144 | Ambiguity, context, route/form, or partial-name cases that should push the search engine toward clarification. |
| `DANGEROUS` | 22,925 | Cases where a confident wrong top-1 can point to a different ingredient or should not produce a confident match. |

Danger is not only copied from the category default. For match rows, `make_case` checks generated collisions and upgrades `SAFE` rows to `CAUTION` or `DANGEROUS` when the query collides with other catalog families, especially different-ingredient families.

## Generation Pipeline

1. `main()` configures logging and output paths.
2. `build_indexed_catalog()` loads `data/canonical_candidates.csv`, creates a `CatalogIndex`, selects usable base records, builds prefix lookup data, and groups product rows by base family.
3. For each `CategoryPlan` in `CATEGORY_PLANS`, `main()` resolves the named generator from `GENERATORS`.
4. The generator creates candidate `GeneratedCase` objects, usually far more than the target count.
5. `select_category_cases()` removes duplicate `(category, input, expected, error_type)` rows, drops unusably short match queries, prioritizes collision cases, and samples deterministically with SHA-256.
6. `assign_difficulties()` assigns exact integer difficulty quotas from the category's difficulty percentages.
7. `validate_final_dataset()` checks total row count, category counts, duplicates, and hard/extreme ratio.
8. The script writes the all-cases CSV, the category summary CSV, and the JSON generation summary.

## Why Deterministic Rule-Based Generation Was Used

The generator is rule-based and deterministic instead of random.

| option considered | why it was not chosen |
| --- | --- |
| Random mutation | It would make the benchmark unstable between runs and harder to debug. A medical search benchmark needs reproducible failures. |
| Manual case lists | Manual lists are easy to miss, hard to scale, and impossible to fully trace. The user required every case to come from code. |
| Rule-based deterministic generation | Chosen because every row gets a category, error type, generator function, source id, collision note, and stable repeatability. |

Deterministic sampling uses `stable_hash()` with the namespace `commercial-name-testing-dataset-v2`. The namespace prevents future generators from accidentally reusing the same hash context for unrelated sampling decisions.

## Category Generator Details

### Product Typo Categories

Categories 1 through 15 are commercial-name mutation cases. They use visual substitutions, phonetic substitutions, multi-character sound rewrites, ligature confusions, deletion, insertion, transposition, truncation, multi-error chains, speed-typing artifacts, vowel guessing, punctuation/copy-paste artifacts, case sensitivity, and number/word confusion.

The most important design choice is that these categories mutate real catalog commercial families. That means the expected target is not synthetic. Every generated typo has a source commercial family and can be tested against the actual search catalog.

### Semi-Outside Context Category

Category 16 mixes a brand fragment with form and strength tokens. This tests a common pharmacy workflow: users rarely type only the brand. They often add `500 mg`, `syrup`, `cream`, `70ml`, or similar context. Algorithm 1 was stronger on these cases than the Algorithm 2 typo-focused approach, which directly influenced Algorithm 3's weights.

### Safety Categories

Categories 17 through 22 are safety-first. They are not just harder typos.

| category | why it matters |
| --- | --- |
| `dangerous_ed1_pairs` | One edit apart can still be a different ingredient. A confident wrong top-1 is unsafe. |
| `substring_traps` | A short real drug can be the prefix of another different drug. Prefix confidence must be conservative. |
| `negative_no_match_expected` | Search should avoid matching random or metadata-like garbage to a real medicine. |
| `contradictory_form_route` | A brand can be real while the requested route/form is inconsistent. |
| `cancelled_na_drug_lookup` | Status-warning rows should not be treated like ordinary safe matches. |
| `score_gap_ambiguity_detection` | Until live score-gap mining is wired in, catalog collision proxies simulate close ambiguous candidates. |

### Algorithmic Robustness Categories

Categories 23 through 30 target failure modes that are easy for one algorithm to miss: OCR digit/letter confusion, OCR plus another error, double-letter expansion/reduction, sampled keyboard adjacency, four-plus error chains, heavy vowel corruption, multi-word fragmentation, and phone autocorrect artifacts.

### Smoke Categories

Categories 31 through 34 are not meant to dominate the benchmark. They exist to catch basic regressions:

1. Exact names should work.
2. Exact names with real strength tokens should work.
3. Whole-word keyboard shift should be measured.
4. Prefix ambiguity should be handled conservatively.

The smoke scope is only 6,000 rows out of 115,000. That prevents exact-match success from inflating the overall score.

## Deduplication And Sampling

`select_category_cases()` deduplicates by:

```text
(category, input, expected, error_type)
```

It does not deduplicate only by `input`, because the same input may be intentionally ambiguous in safety categories. It also does not deduplicate only by `expected`, because many different mutations should map to the same family.

After deduplication, rows are sorted by:

1. Collision presence first.
2. Stable SHA-256 hash of namespace, category, input, expected, and error type.

Collision-first ordering makes the benchmark stricter. It favors cases where the search engine must choose or clarify among possible families.

## Validation Rules

The generator fails loudly instead of silently producing a partial dataset.

| validation | consequence if it fails |
| --- | --- |
| Every category generator exists in `GENERATORS`. | Raises `ValueError`. |
| Every category produces at least its target count after deduplication. | Raises `RuntimeError`. |
| Every selected category count equals the plan target. | Raises `RuntimeError`. |
| Total rows equal the sum of explicit category targets. | Raises in `validate_final_dataset()`. |
| Duplicate final keys are not allowed. | Raises in `validate_final_dataset()`. |
| Hard/extreme ratio must be at least 60%. | Raises in `validate_final_dataset()`. |
| Empty match queries are not allowed. | Raises in `make_case()` or `normalize_query_for_output()`. |

This is intentional. In a medical search benchmark, silently missing a category would make later accuracy numbers misleading.

## How To Regenerate

Run from `medicine-search-clean`:

```bash
PYTHONDONTWRITEBYTECODE=1 python3 'benchmark_02_synthetic/generate_dataset.py'
```

The expected regenerated outputs are:

```text
benchmark_02_synthetic/data/test_cases.csv
benchmark_02_synthetic/data/category_summary.csv
benchmark_02_synthetic/data/generation_summary.json
```

If the source catalog changes, generated examples and counts before sampling may change. The script will still enforce the 34 planned category targets unless the catalog no longer has enough candidates.
